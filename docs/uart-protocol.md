# UART Protocol

TangMiner exposes a small binary UART protocol so host software can treat the
FPGA as a nonce-scanning hash engine.

UART settings:

- `115200`
- `8N1`
- no flow control

All job fields are sent as raw bytes. SHA-256 words are big-endian. The returned
nonce is the four-byte wire-order value inserted into Bitcoin header bytes
`76..79`.

## Command Framing

Every host command starts with:

```text
"T" "N" command
```

Unknown commands are ignored and reset the parser to sync search.

## Start Job

Host to FPGA:

```text
"T" "N" "J"
midstate[32]
tail[12]
target[32]
```

Total length: `79` bytes.

Fields:

- `midstate`: SHA-256 internal state after the first 64 bytes of the Bitcoin
  block header.
- `tail`: header bytes `64..75`, excluding the nonce. These are interpreted as
  three big-endian SHA-256 message words.
- `target`: 32-byte big-endian proof-of-work target, or one of the recognized
  alias values generated by the repo tools.

The FPGA constructs the first-pass final block as:

```text
tail[12] || nonce[4] || 0x80 || zero padding || 0x00000280
```

It then runs the second SHA-256 pass over the first digest. The current 20K
bitstream starts five lanes at nonce words `0` through `4`; each lane
increments by `5`.

## Candidate Filtering

The 20K design does not implement a full 256-bit FPGA target
comparator. Instead, recognized target values select small prefix filters over
`reverse_bytes(hash)`, and the host performs the authoritative target check.

Recognized aliases:

- `all-ones` / `easy`: always report the first checked nonce.
- `quick3`: require the top 3 bits of `reverse_bytes(hash)` to be zero.
- `quick21`: require the top 21 bits of `reverse_bytes(hash)` to be zero.
- `quick14`: require the top 14 bits of `reverse_bytes(hash)` to be zero.
- `quick23`: require the top 23 bits of `reverse_bytes(hash)` to be zero.
- `quick26`: require the top 26 bits of `reverse_bytes(hash)` to be zero.

On the default 5-lane `54.000 MHz` 20K build:

```text
quick21: about 0.50 s per candidate
quick23: about 2.0 s per candidate
quick26: about 16.0 s per candidate
```

Arbitrary 32-byte target values currently select the `quick23` hardware
candidate filter. This keeps the FPGA comparator small while still letting host
software validate exact shares.

## Found Response

FPGA to host:

```text
"F"
nonce[4]
```

Total length: `5` bytes.

The FPGA does not return the hash. Host software must rebuild the 80-byte
header, copy the nonce bytes into header bytes `76..79`, double-hash the header,
and compare the resulting proof-of-work integer against the actual share target.

## Echo Job

Host to FPGA:

```text
"T" "N" "E"
midstate[32]
tail[12]
target[32]
```

FPGA to host:

```text
"E"
midstate[32]
tail[12]
target[32]
```

Total response length: `77` bytes.

This command does not start hashing. It is useful for checking serial wiring,
parser alignment, and byte order.

## Stop Job

Host to FPGA:

```text
"T" "N" "S"
```

This stops the current scan and returns the core to idle. There is no
acknowledgement response.

## Hardcoded Test Job

Host to FPGA:

```text
"T" "N" "H"
```

This starts a built-in genesis-style easy-target test job. It exists for
bring-up; miner host software should use `TNJ`.

## Make A Job Packet

```sh
python3 scripts/tools/make_job.py \
  --header <80-byte-header-hex> \
  --target <32-byte-big-endian-target-hex|quick26|quick23|quick21|quick3|all-ones> \
  > job.bin
```

Send `job.bin` to the board over USB-UART, or use it as a fixture for host
driver tests.

## Mujina Integration Sketch

A Mujina backend can sit at the same layer as an ASIC hashboard driver:

1. Convert pool work into a block header and expanded target.
2. Precompute the SHA-256 midstate for bytes `0..63` of the header.
3. Send `TNJ` packets with unique work, such as distinct extranonce2 or merkle
   roots.
4. Listen for `F || nonce` responses.
5. Reconstruct, hash, validate, and submit shares using the nonce endian form
   expected by the pool stack.

See [mujina-integration.md](mujina-integration.md) for backend notes.
